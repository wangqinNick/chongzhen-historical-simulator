# SillyTavern 导入与运行指南

这份指南面向最终使用，不要求理解全部 41 个模块。目标是让你能在 SillyTavern 里快速开局，并知道什么时候该导入哪个文件。

## 三种使用方式

### A. 最省事：一键复制版

打开 `sillytavern/最终一键复制包_崇祯历史模拟器.md`，按文件里的分段复制到 SillyTavern。

适合：

- 不想处理多个 Lorebook 文件。
- 只想先开一局测试手感。
- 使用的模型上下文较大。

注意：一键复制包很长，如果模型上下文较小，优先改用 B 方案。

### B. 推荐：角色卡加完整 Lorebook

1. 导入 `sillytavern/崇祯模拟器_Narrator角色卡.json`。
2. 导入 `sillytavern/import/崇祯历史模拟器_完整Lorebook.json`。
3. 在当前聊天的 Author's Note 里粘贴一个开局存档，例如 `saves/开局存档/1628_崇祯元年.md`。
4. 第一回合按诏书格式输入军事、内政、财政、外交、人事命令。

适合：

- 想长期玩一个存档。
- 希望 Lorebook 按触发词加载，不把全部资料硬塞进上下文。
- 之后想逐步加入人物群聊。

### C. 轻量：核心规则加状态快照

1. 导入 `sillytavern/崇祯模拟器_Narrator角色卡.json`。
2. 只导入 `sillytavern/import/崇祯历史模拟器_核心Lorebook.json`。
3. 把 `sillytavern/当前状态_AuthorNote模板.md` 或某个开局存档放进 Author's Note。
4. 需要人物、地区、军队时，再追加导入对应的 Lorebook。

适合：

- 模型上下文较小。
- 先测试裁判逻辑和回合结构。
- 想自己逐步扩展人物和地区。

## Lorebook 选择

`崇祯历史模拟器_完整Lorebook.json` 包含核心规则、41 个模块、快速卡、开局剧本和首批人物条目，是最推荐的单文件导入方案。

`崇祯历史模拟器_核心Lorebook.json` 只保留裁判、财政、军队、灾荒、科技、状态快照、诏书等高频规则，适合轻量开局。

`崇祯历史模拟器_全模块Lorebook.json` 包含 41 个模块，但不含人物快速卡，适合只想跑制度系统的局。

`崇祯历史模拟器_人物Lorebook.json` 把 20 个首批人物做成 Lorebook 条目，适合单 Narrator 模式。

`崇祯历史模拟器_快速卡Lorebook.json` 包含地区、军队、派系快速卡，适合补充局势常识。

`崇祯历史模拟器_开局剧本Lorebook.json` 包含 6 个年份的开局 Author's Note，可用于切换开局。

## 快速卡怎么用

快速卡适合放入 Data Bank / RAG，也可以导入 `崇祯历史模拟器_快速卡Lorebook.json`。它们不是完整数据库，而是高频触发的局势压缩卡。

当前快速卡包括京师、辽东、陕西、河南、江南、山东登莱、宣大蓟镇、南京南直隶、湖广、四川、山西、闽粤海贸，以及关宁军、京营、卫所、剿贼官军、东林士论、阉党清算、商帮海商、漕运、宗藩、驿传塘报等节点。

如果你使用完整 Lorebook，这些快速卡已经合并进去；如果模型上下文较小，可以只导入核心 Lorebook，再把快速卡 Markdown 放进 Data Bank。

## 人物卡怎么用

如果只用单 Narrator，不需要导入人物角色卡，导入人物 Lorebook 就够了。

如果要 Group Chat，可逐个导入 `sillytavern/人物卡/json/` 下的角色卡。建议先只加入 3 到 5 人，例如崇祯、温体仁、毕自严、孙承宗、袁崇焕。等状态稳定后，再加入皇太极、李自成、徐光启、孙元化等人物。

多角色群聊更有戏剧性，但也更容易消耗上下文。长期存档仍应以 Author's Note 的状态快照为准。

## 开局存档

`saves/开局存档/` 提供 6 个入口：

- 1628 崇祯元年：适合从继位初期重整朝局。
- 1629 崇祯二年：适合处理己巳之变前后辽东与京师压力。
- 1631 崇祯四年：适合处理辽东、登莱、西学火器和财政连锁。
- 1634 崇祯七年：适合处理流民、剿抚、军饷和地方治理。
- 1637 崇祯十年：适合处理多线战争和财政疲劳。
- 1641 崇祯十四年：适合高难度救火局。

每个存档都有 Markdown 和 JSON 两种形式。Markdown 便于复制到 Author's Note，JSON 便于二次加工。

## 每回合推荐流程

1. 玩家输入诏书，至少包含政策名称、主办官、资金来源、执行期限、验收标准。
2. Narrator 按财政、军队、地方、派系、合法性、信息误差逐项校验。
3. Narrator 输出奏报、执行结果、风险、人物反应和下回合建议。
4. 回合末更新状态快照。
5. 只有会影响未来决策的变化进入长期状态。

## 常见问题

如果模型忘记局势，把当前状态快照重新贴到 Author's Note，并要求“以 Author's Note 为最高当前状态”。

如果 Lorebook 触发过多，改用核心 Lorebook，并把人物、地区、军队卡放进 Data Bank / RAG。

如果人物说话抢戏，减少 Group Chat 角色数量，让 Narrator 继续做总裁判。

如果政策总是成功，提醒 Narrator 必须执行财政、资源、交通、地方阻力、派系反噬、信息误差六项检查。

如果输出太散，要求固定使用 `modules/17_诏书_政策_回合总结输出模板.md` 的回合格式。
