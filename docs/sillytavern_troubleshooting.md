# SillyTavern 排错手册

这份手册处理实际导入和游玩时最常见的问题。

## 角色卡导入后不像裁判

现象：

- 模型像大臣一样说话。
- 模型替玩家下决定。
- 模型只讲剧情，不做执行检查。

处理：

1. 确认导入的是 `sillytavern/崇祯模拟器_Narrator角色卡.json`。
2. 在当前聊天发送：

```text
你是“大明国运裁判”，不是单一臣子。玩家才是崇祯皇帝。请恢复裁判模式：不替玩家决策，每条政策必须检查财政、资源、执行人、地方、交通、派系、合法性、信息误差和敌方行动。
```

## Lorebook 触发太多

现象：

- 回复很长。
- 每回合都引用大量无关模块。
- 模型变慢或忘记当前状态。

处理：

- 改用 `崇祯历史模拟器_核心Lorebook.json`。
- 把快速卡和人物卡放进 Data Bank / RAG。
- 只在相关剧本中导入人物 Lorebook。
- 避免在用户输入中反复使用“大明”“崇祯”这类过宽触发词。

## Lorebook 触发太少

现象：

- 模型忘记财政、军队、灾荒等规则。
- 输出变成普通历史聊天。

处理：

1. 导入 `崇祯历史模拟器_完整Lorebook.json`。
2. 在玩家指令中明确触发关键词，例如“财政、国库、辽饷、军饷、粮道、灾荒、民变、后金、流民军”。
3. 把 `sillytavern/玩家指令模板.md` 中的严格模式复制到聊天框。

## Author's Note 漂移

现象：

- 旧状态和新状态混在一起。
- 已经死亡、被贬、被俘的人物又恢复原状。
- 年份错乱。

处理：

发送：

```text
以下是新的当前状态快照，请用它覆盖旧状态。若旧聊天记录与此冲突，以这份 Author's Note 为准。请确认覆盖后再继续。

【粘贴新的 Author's Note】
```

长期建议：

- 每回合只更新 Author's Note 的变化项。
- 使用 `docs/state_field_dictionary.md` 统一字段含义。
- 使用 `saves/回合日志/回合日志_可复制模板.md` 另存摘要。

## 政策总是成功

现象：

- 玩家一下诏，政策就完成。
- 没有财政、粮道、地方执行问题。
- 敌方没有反应。

处理：

发送：

```text
进入严格历史裁判模式。不得默认政策成功。每条政策必须输出：可执行性检查、资源瓶颈、执行阻力、敌方自动行动、失败路径和反噬风险。
```

如果仍然过于顺利，要求模型把难度切换为“史实高压”。

## 群聊人物抢戏

现象：

- 所有人都发言。
- 人物互相聊天，Narrator 不总结。
- 回合状态没有更新。

处理：

- 按 `sillytavern/群聊预设/推荐群聊组合.md` 只导入 3 到 7 人。
- 每回合只点名 3 到 5 个相关人物。
- 要求 Narrator 最后总结并更新 Author's Note。

可发送：

```text
本回合只允许相关人物发言。无关人物静默。所有人物发言后，由 Narrator 统一裁判、结算并给出更新后的 Author's Note。
```

## 历史依据被说得太确定

现象：

- 网络索引被写成确定史实。
- 人物能力值像史学结论。
- 争议事件没有不确定性。

处理：

- 导入 `崇祯历史模拟器_资料规则Lorebook.json`。
- 把 `research/source_reliability_guide.md` 放进 Data Bank。
- 要求输出依据类型。

可发送：

```text
请按资料可信度分层重写上一段：史实依据、学术整理、网络索引、合理推断、模拟器采用值分别列出。不得把网络索引或模拟器数值写成史学定论。
```

## 最终复制包太长

现象：

- 模型上下文不够。
- 导入后回复变慢。

处理：

用轻量方案：

1. 角色卡：`崇祯模拟器_Narrator角色卡.json`
2. Lorebook：`崇祯历史模拟器_核心Lorebook.json`
3. Author's Note：`saves/开局存档/` 中的某个年份
4. Data Bank：快速卡、资料可信度指南、状态字段字典

## 发布包完整性

运行：

```powershell
python scripts\check_release_bundle.py
```

应看到：

```text
OK: release bundle contains 32 character cards, 7 opening saves, 7 lorebooks
```
